# Звіт: Покриття вакансії "Lead Full Stack AI Developer" проєктом notesQa

> **Роль:** Principal AI Architect / Technical Interviewer
> **Кандидат:** Ihor 
> **Проєкт-доказ:** notesQa (NestJS, LangGraph, OpenSearch, Redis Stack, WebSockets)

Цей документ є об'єктивним аудитом твого пет-проєкту `notesQa` на відповідність вимогам вакансії Lead AI Engineer. Він створений як стратегічна шпаргалка для підготовки до технічної та менеджерської співбесіди.

## 1. Загальний відсоток покриття: ~85%

**Вердикт:** Твій технічний бекграунд, підтверджений кодовою базою `notesQa`, є **надзвичайно сильним** для позиції Lead AI Engineer. Ти закриваєш абсолютно всі вимоги щодо розробки агентів (графічна архітектура, пам'ять, RAG, WebSockets). 

Проєкт виходить далеко за рамки типових "пет-проєктів" (де зазвичай просто смикають OpenAI API) і демонструє production-ready мислення (DI, Fallbacks, Tracing, Hybrid Search, Human-in-the-loop). 

Залишкові 15% — це інфраструктурні (AWS, Microservices) та менеджерські (управління реальною командою) навички, які легко "захистити" на співбесіді.

## 2. Мапінг: Вимога ↔ Реалізація в коді (Покрито на 100%)

Ось як ти маєш відповідати на запитання по кожному пункту вакансії, спираючись на конкретний код:

### 🎯 Agent-workflow (планування, інструменти, пам’ять, самокорекція)
**Твій доказ:** Топологія `StateGraph` у `agent.graph.ts`.
- **Планування:** Реалізовано через `planner` ноду (gpt-4o-mini), яка визначає стратегію (`search`, `tools`, `direct`).
- **Пам'ять:** Унікальна 3-рівнева архітектура (`MemoryOrchestratorService`). STM (вікно контексту) + Episodic LTM (Redis Stack JSON) + Semantic (OpenSearch). 
- **Самокорекція:** `grader.node.ts` валідує відповідь генератора. Якщо є галюцинації — ініціюється внутрішній цикл перегенерації (атрибут `retryCount` у `AgentState` запобігає нескінченним циклам).
- **Інструменти:** Динамічний бандінг інструментів (`llm.bindTools()`) через `createAgentTools`.

### 🎯 LangGraph & LangChain на production-рівні
**Твій доказ:**
- Вирішення проблеми **Language Drift**: Жорстке закріплення мови через промпти та системні повідомлення на потрібних етапах графа.
- Вирішення проблеми **Redis Stack**: Оновлення Docker-контейнера та налаштування кастомного `checkpointer` для `LangGraph` (вирішення помилок `JSON.SET` та `FT.CREATE` при персистенції стану графа).
- **WebSockets + Interrupts:** Перехоплення стану графа (`__interrupt__`) через `AgentService.stream()` та еміт подій через `AgentEventEmitter` (Mediator pattern) для реалізації Human-in-the-loop, не ламаючи з'єднання.
- **Provider Fallbacks:** Використання `RunnableWithFallbacks` в `LlmService` (автоматичний перехід з `gpt-4o` на `gpt-4o-mini` при падінні чи rate-лімітах).

### 🎯 RAG-рішення та тестування AI-систем
**Твій доказ:**
- **Agentic RAG:** Замість прямого пошуку реалізовано `query_rewriter` ноду, яка бере історію та переписує питання користувача в ідеальний пошуковий запит.
- **Тестування:** Файли `agent-routing.spec.ts` (детерміноване тестування роутерів графа) та `agent-evaluation.spec.ts` (патерн LLM-as-a-Judge, шаблони тестування на галюцинації).

### 🎯 TypeScript, Node.js та патерни проєктування
**Твій доказ:** 
- Жорстка типізація `AgentState` через `Annotation.Root`.
- Інтеграція AI-фреймворку в **IoC / DI контейнер NestJS** (чіткий розподіл на `LlmModule`, `AgentModule`, `MemoryModule`). 
- Патерни: **Strategy** (різні маршрути роутера), **Mediator** (EventEmitter для WebSockets), **Fallback** (LLM resilience).

### 🎯 Векторні БД (OpenSearch)
**Твій доказ:**
- Реалізовано **Hybrid Search**: Комбінація повнотекстового пошуку (BM25 - `match`) та векторного семантичного пошуку (kNN) через Elasticsearch `bool should` запит у `OpenSearchRepository`.

   - *Факт:* Проєкт — це моноліт на NestJS, що запускається через локальний `docker-compose` з OpenAI API.

3. **Реальне "People Management" (Онбординг, Менеджмент команди):**
   - *Факт:* Код пишеш ти один. Конфлікти і перформанс-рев'ю в пет-проєктах не трапляються.

## 4. Стратегія захисту (Pitch Strategy)

Як перетворити прогалини на сильні сторони під час співбесіди:

### Аргумент 1: "У вас немає досвіду з AWS Bedrock, тільки OpenAI"
**Твоя відповідь:** 
> "Так, у своєму проєкті notesQa я підняв все локально (OpenAI + Redis container) для швидкості розробки. Але ключова фішка LangChain та моєї архітектури на NestJS полягає в абстракціях. У моєму `LlmModule` перехід з OpenAI на AWS Bedrock — це заміна одного класу з `ChatOpenAI` на `ChatBedrock`. Граф, роутери та пам'ять навіть не дізнаються про цю зміну, бо вони залежать від інтерфейсу `BaseChatModel`. Окрім того, я спеціально реалізував патерн `withFallbacks`, який дозволяє комбінувати провайдерів (наприклад, Bedrock як primary, OpenAI як резерв)."

### Аргумент 2: "Ми шукаємо людину для мікросервісної архітектури на AWS. У вас моноліт"
**Твоя відповідь:**
> "Мій проєкт notesQa спроєктований як модульний моноліт саме для того, щоб його було легко рознести на мікросервіси в AWS. Інтерфейси чітко розділені.
> **Мій план міграції на AWS такий:**
> 1. Compute: Перенесення NestJS у Docker контейнери на **AWS ECS (Fargate)**.
> 2. WebSocket: Нода, що тримає WebSockets і AgentService, може бути винесена в окремий сервіс або використовувати API Gateway WebSocket API + **AWS Lambda** для stateless викликів графа.
> 3. Обробка файлів: Додавання документів у базу — через завантаження в **AWS S3**, що тригерить подію на векторне вбудовування.
> 4. Бази даних: Замість локального Redis — **Amazon ElastiCache**, замість локального OpenSearch — **AWS OpenSearch Service/Serverless**."

### Аргумент 3: "У вас є досвід створення Roadmap і управління ризиками?" (Вимоги 10 та 11)
**Твоя відповідь:**
> "Так, я підходжу до розробки проактивно. Створення складної системи з WebSockets та Human-in-the-loop вимагало серйозного планування. В рамках свого проєкту я створив ряд документів, які є стандартом для Team Lead:
> - **`ai-lead-roadmap.md`** — переклад продуктових цілей (додавання пам'яті, агентності) в технічні етапи.
> - **`ws-fix-plan.md`** — приклад аналізу вузького місця (коли переривання графа ламало UI) та покрокового плану вирішення без даунтайму.
> - **`production-hardening-plan.md`** — мій проактивний підхід до ризиків. Я не чекав, поки система впаде в продакшені, а заздалегідь спроєктував Observability (LangSmith), гібридний пошук для підвищення recall та Fallbacks для resilience."

### Аргумент 4: "Як ви тестуєте недетерміновані системи?"
**Твоя відповідь:**
> "У мене комбінований підхід. У своєму проєкті я розділив логіку:
> 1. Ноди-роутери (планувальник, грейдер) перевіряються класичними детермінованими unit-тестами з моками (Jest), перевіряючи edge-кейси (LLM видав галюцинацію, перевищено retryCount).
> 2. Сама генерація (RAG) перевіряється через патерн **LLM-as-a-Judge**. Я створив інфраструктуру, де тестовий фреймворк відправляє питання, отримує відповідь генератора, і передає їх 'моделі-судді' з промптом, який перевіряє факт галюцинації або релевантності. В продакшені я б зв'язав ці тести з LangSmith Evaluation для автоматизації при кожному PR."
