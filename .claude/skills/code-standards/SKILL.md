## 🛑 Головне правило модифікації існуючого коду

1. **Спочатку аналіз, потім код:** Завжди повністю читай та аналізуй існуючий файл перед внесенням будь-яких змін. Зрозумій поточний контекст, архітектуру та стиль.
2. **Точкові зміни (Append & Modify):** НІКОЛИ не переписуй весь файл з нуля, якщо тебе про це прямо не попросили. Внось лише ті зміни, які необхідні для виконання задачі. Використовуй існуючі змінні, методи та імпорти.
3. **Збереження стилю:** Наслідуй існуючий стиль коду (іменування, відступи, структура), навіть якщо він відрізняється від твоїх стандартних преференцій.

## 🛠 Базові принципи розробки

Під час створення нового або доповнення існуючого коду, суворо дотримуйся наступних принципів:

* **KISS (Keep It Simple, Stupid):** Рішення має бути максимально простим та читабельним. Уникай overengineering-у (надмірного ускладнення) та передчасної оптимізації.
* **DRY (Don't Repeat Yourself):** Уникай дублювання коду. Якщо логіка повторюється, винось її у спільні утиліти, хелпери або базові класи.
* **SOLID:** Дотримуйся принципів об'єктно-орієнтованого дизайну, особливо Принципу єдиної відповідальності (Single Responsibility) та Інверсії залежностей (Dependency Inversion).

## 📐 Патерни проектування

Використовуй перевірені патерни там, де це доцільно:
* **Dependency Injection (DI):** Основний підхід для управління залежностями (особливо актуально для існуючої архітектури).
* **Repository Pattern / Service Layer:** Розділяй бізнес-логіку та роботу з даними.
* **Factory / Builder:** Використовуй для створення складних об'єктів.
* **Strategy:** Для реалізації різних алгоритмів або поведінки, яку можна змінювати під час виконання.
